What Is the Internet, Physically?

The internet is not a cloud. It is wires and machines. Specifically: fiber optic cables running under oceans and across continents, routers forwarding packets toward their destination, data centers housing thousands of servers, and your home router as the last hop between you and everything else.

0.2

Packets: How Data Actually Travels

When you send or receive data, it is not sent as one big chunk. It is sliced into small pieces called packets — typically around 1,500 bytes each. Why? Because smaller pieces can take different routes through the network, lost pieces can be resent individually, and multiple users can share the same wire fairly.

You download a 3 MB image.
The image is split into ~2,000 packets.

Each packet travels independently:
  Packet 1  →  Router A → Router C → Router F → Destination
  Packet 2  →  Router A → Router B → Router D → Destination
  Packet 3  →  Router A → Router C → Router E → Destination

They may arrive out of order.
TCP reassembles them into the correct sequence.
ProtocolGuaranteesUse case
TCPAll packets arrive, in the correct order. Lost packets are re-requested.Web pages, API calls, file downloads — anything where correctness matters.
UDPNo guarantees. Fire and forget.Video calls, online games — where a dropped frame is better than freezing.
🔑
Key insight TCP is the reliability layer. HTTPS uses TCP underneath. Every web request you make is TCP — the browser and server silently negotiate packet delivery, retransmission, and ordering so you never have to think about it.
0.3

DNS: The Internet's Phone Book

Humans remember github.com. Machines only understand 140.82.112.4. DNS (Domain Name System) is a globally distributed database that translates one to the other.

When you type github.com and press Enter, this chain executes:

Your Browser
    │
    │  1. Check local DNS cache — "Have I looked this up recently?"
    │     → No (or TTL expired)
    ▼
Your Operating System
    │
    │  2. Check /etc/hosts — a local override file
    │     → Not there
    ▼
Your ISP's DNS Resolver  (e.g. 8.8.8.8 = Google's public DNS)
    │
    │  3. Check its cache → Not there
    ▼
Root Name Server  (13 server clusters, worldwide)
    │
    │  4. "I don't know github.com, but I know who handles .com"
    │     → Returns address of TLD nameserver
    ▼
TLD Name Server  (.com registry, run by Verisign)
    │
    │  5. "GitHub's authoritative nameserver is ns1.github.com"
    ▼
GitHub's Authoritative DNS Server
    │
    │  6. "github.com → 140.82.112.4"
    │     TTL: cache this result for 60 seconds
    ▼
Back to your browser — connection begins

The full chain takes 20–120 ms the first time. After that, results are cached and future lookups are nearly instant.

0.4

TCP Connection: The Handshake

After DNS resolves the IP, the browser needs to establish a reliable connection before sending any data. This is the TCP 3-way handshake:

Browser                              Server (140.82.112.4:443)
   │                                       │
   │  ── SYN ──────────────────────────►  │   "I want to connect"
   │                                       │
   │  ◄── SYN-ACK ───────────────────────  │   "Received. I'm ready."
   │                                       │
   │  ── ACK ──────────────────────────►  │   "Acknowledged."
   │                                       │
   │  [TCP connection established]         │
   │                                       │
   │  [TLS handshake begins — for HTTPS]   │
   │  Exchange encryption keys,            │
   │  verify server certificate            │
   │                                       │
   │  [HTTP request goes here]             │

For HTTPS, a TLS handshake follows the TCP handshake. The two sides agree on an encryption algorithm, exchange cryptographic keys, and verify the server's identity via its SSL certificate (signed by a trusted Certificate Authority). After that, all data is encrypted — nobody intercepting packets in between can read anything useful.

ℹ️
Port numbers The server listens on port 443 for HTTPS and port 80 for plain HTTP. A port is just a number that identifies which program on the machine should receive the traffic — like an apartment number in a building. Your Node.js apps will typically use port 3000 in development.

What a Browser Actually Does

A browser is not just a window that displays things. It is a rendering engine + a JavaScript engine + a networking layer. When it receives HTML from the server, this pipeline runs:

HTML text received from server
        │
        ▼
┌─────────────────────────┐
│      HTML Parser        │  Reads characters one by one.
│                         │  Builds a tree of nodes in memory.
└────────────┬────────────┘
             │                    ← Meanwhile: parser hits <link rel="stylesheet">
             │                      → fires HTTP GET for style.css
             │                      → builds CSSOM (CSS Object Model)
             ▼
┌─────────────────────────┐
│        DOM Tree         │  Document Object Model
│                         │  Every HTML element = a node
│   <html>                │
│   ├── <head>            │
│   │   └── <title>       │
│   └── <body>            │
│       ├── <h1>          │
│       └── <p>           │
└────────────┬────────────┘
             │
             │  DOM + CSSOM merged
             ▼
┌─────────────────────────┐
│      Render Tree        │  Only visible nodes (display:none excluded)
│                         │  Each node has its computed styles attached
└────────────┬────────────┘
             ▼
┌─────────────────────────┐
│         Layout          │  Calculate exact pixel position + size
│        (Reflow)         │  of every element on screen
└────────────┬────────────┘
             ▼
┌─────────────────────────┐
│          Paint          │  Fill pixels: backgrounds, borders, text
└────────────┬────────────┘
             ▼
┌─────────────────────────┐

│                         │  Send final image to GPU → screen
└─────────────────────────┘
⚡
Why this matters for performance Changing CSS width triggers Layout → Paint → Composite (expensive). Changing color skips Layout (cheaper). Changing transform or opacity skips both Layout and Paint — the GPU handles it alone (cheapest). This is why CSS animations using transform are smooth, while those using width or top can be janky.

Website Anatomy: The Full Stack

Every web application is made of at most four layers. Understanding what each layer does — and crucially, what it does not do — is the most important architectural concept in web development.

🔑
The cardinal rule of web architecture The browser never talks to the database directly. The server is the gatekeeper. This is not a performance choice — it is a security requirement. If the browser could query the database directly, any user could read any other user's data.

Full Request Lifecycle — End to End

Let's trace every single thing that happens when you click the Like button on a tweet. Each step is real. Total round-trip time: 50–200 ms.

Color shows where each step runs: ■ Browser ■ Network ■ Server ■ Database
1
Click event fires Browser

The browser detects a mouse click on #like-button. The JavaScript event listener registered with addEventListener runs immediately — no network involved yet.

2
JS builds an HTTP request Browser

fetch() is called with the URL, method POST, your login token in the Authorization header, and the tweet ID as a JSON body. The browser packages this into an HTTP frame ready to send.

3
DNS lookup Network

Browser checks its local DNS cache — api.twitter.com was resolved recently. IP address 104.244.42.129 found instantly. The full DNS chain (Lesson 0.3) is skipped.

4
TCP + TLS connection Network

Browser reuses an already-open TCP connection to Twitter's server (HTTP/2 multiplexing allows multiple requests to share one connection). No new handshake needed — saves ~100 ms.

5
Request travels the internet Network

The encrypted HTTP/2 frame leaves your machine, hops through routers across the internet, and arrives at Twitter's data center. Takes roughly 10–80 ms depending on geography.

6
Server receives & routes the request Server

The Express router matches the pattern POST /2/tweets/:id/like. Auth middleware runs first: it decodes the JWT Authorization token and extracts user_id = 9999. Then the route handler is called.

7
Database write Database

Server sends two SQL queries: an INSERT into the likes table (user 9999 liked tweet 12345), and an UPDATE on the tweet row to increment like_count by 1. Database confirms success.

8
Server builds the response Server

Handler serializes the result to JSON: {"liked": true, "like_count": 42301}. Sets HTTP status 200 OK and Content-Type header, then sends it back through the open connection.

9
Response travels back Network

The encrypted response frames hop back across the internet and arrive in your browser's receive buffer. Same route, opposite direction.

10
JS callback runs — UI updates Browser

The fetch() Promise resolves. Your callback reads the JSON body, writes 42301 into the like count element, and adds a CSS class to turn the heart red. The browser repaints only those changed nodes — the rest of the page is untouched.

The JavaScript that drives all 10 steps
document.querySelector('#like-button').addEventListener('click', async () => {

  // Steps 1–2: event fires, fetch() builds and sends the HTTP request
  const response = await fetch('https://api.twitter.com/2/tweets/12345/like', {
    method: 'POST',
    headers: {
      'Authorization': 'Bearer eyJhbGc...',  // your login token
      'Content-Type':  'application/json'
    },
    body: JSON.stringify({ tweet_id: '12345' })
  });

  // Steps 3–9 happen invisibly while await suspends this function:
  //   DNS lookup → TCP connection → Network → Server → Database → Network

  // Step 10: response arrived — update the DOM
  const data = await response.json();
  document.querySelector('#like-count').textContent = data.like_count;  // 42301
  document.querySelector('#like-button').classList.add('liked');         // heart → red
});
🔑
The key takeaway The browser and server are in a constant conversation — request and response, back and forth. The browser never knows what is in the database; it only knows what the server chooses to send it. Every modern web app you use runs this exact loop, thousands of times per minute.

How CSS Works: Selectors, Cascade & Specificity

CSS has three interlocking systems you must understand before anything else: selectors (which elements to style), the cascade (which rule wins when multiple apply), and inheritance (which properties flow from parent to child).

p              { }   /* all <p> elements */
.card          { }   /* elements with class="card" */
#main-nav      { }   /* element with id="main-nav" — only one */
.card p        { }   /* <p> anywhere inside .card */
.card > p      { }   /* <p> that is a direct child of .card */
a:hover        { }   /* <a> when mouse is over it */
input:focus    { }   /* <input> when it has keyboard focus */
p::first-line  { }   /* the first rendered line of every <p> */

When multiple rules target the same element, specificity determines the winner. Each selector type earns points:

Priority (highest wins first, then points, then order):

  1.  !important           →  nuclear override — avoid
  2.  Inline style         →  style="color:red"
  3.  ID selector          →  #nav             = 100 pts
  4.  Class / pseudo-class →  .card   :hover   =  10 pts
  5.  Element selector     →  p   h1           =   1 pt
  6.  Last-written wins    →  when points are equal

Example:
  #nav .item a:hover   =  100 + 10 + 1 + 10  =  121 pts  ← wins
  .nav-item            =  10                  =   10 pts

Async JS & fetch()

JavaScript is single-threaded — one thing runs at a time. A network request can take 100–300 ms. If JS waited synchronously, the page would freeze solid. The solution: JavaScript delegates the waiting to the browser, keeps running other code, and picks up the result when it's ready.

Call Stack            Browser (Web APIs)         Task Queue
──────────            ──────────────────         ──────────
loadUsers()  →   fetch('/api/users')
             →   [network request running…]
main exits                              →    [callback ready]
                                                    │
             ←── call stack picks it up when empty ─┘
// async/await — the clean, modern way to handle async operations
async function loadUsers() {
  try {
    const response = await fetch('/api/users');  // suspend here (non-blocking)
    const users    = await response.json();      // suspend until body is parsed
    return users;
  } catch (error) {
    console.error('Failed:', error);
  }
}

// await only suspends this function — the rest of the page keeps running
loadUsers();
// Checking the response before using it
const response = await fetch('/api/users');

response.ok;      // true if status 200–299
response.status;  // the exact code: 200, 404, 500, etc.

// Parse the body — only call ONE of these, only ONCE per response:
const data = await response.json();   // parse JSON
const text = await response.text();   // raw text

});

2.1

The HTML Parser

The browser doesn't wait for the full HTML file to arrive before it starts working. It parses incrementally — processing bytes as they stream in from the network, building the DOM in real time.

Two stages: Tokenization → Tree Construction

Raw bytes arrive:  3C 68 31 3E 48 65 6C 6C 6F 3C 2F 68 31 3E
Decoded text:      < h  1  >  H  e  l  l  o  <  /  h  1  >

Tokenizer produces:
  StartTag token  →  { name: "h1" }
  Character token →  { data: "Hello" }
  EndTag token    →  { name: "h1" }

Tree builder turns tokens into DOM nodes:
  Element: HTMLHeadingElement (h1)
    └── Text: "Hello"

The parser is intentionally error-tolerant. Missing closing tags, overlapping tags, illegal nesting — the browser fixes them silently following a defined error-recovery algorithm. That's why broken HTML still renders.

Render-blocking resources — the most important performance concept in HTML loading:

<!-- Blocks RENDERING — browser won't paint until this is downloaded + parsed -->
<link rel="stylesheet" href="style.css">

<!-- Blocks PARSING — synchronous JS can rewrite the DOM, so parser must stop -->
<script src="app.js"></script>

<!-- Non-blocking — deferred until after HTML parsing; executes in order -->
<script src="app.js" defer></script>

<!-- Non-blocking — loads in parallel; executes immediately when ready -->
<script src="analytics.js" async></script>

The preload scanner is a second lightweight parser that runs ahead through the HTML while the main parser is blocked. It finds <link>, <img>, and <script> tags and fires early HTTP requests so resources are already downloading when the main parser unblocks.

CSS Parsing & The CSSOM

After CSS is downloaded, it is parsed into a CSSOM (CSS Object Model) — a tree that mirrors the DOM structure but holds style information instead of content.

Raw CSS text
    │
    ▼  Tokenizer → Parser
StyleSheet
  ├── Rule: h1         →  { color: red;   font-size: 2rem  }
  ├── Rule: .card      →  { background: white;  padding: 24px }
  └── Rule: .card h1   →  { color: blue }   ← overrides h1 when inside .card

Selector matching runs right-to-left. This is a deliberate performance decision:

CSS rule:  .card h1 { color: blue; }

Browser reads this as:
  "Find all <h1> elements..."          ← start from rightmost key selector
  "...that have an ancestor with class .card"

Why right-to-left?
  Starting from the key selector (h1) immediately filters the full element set.
  Starting from the left (.card) would require walking all descendants first.

Computed styles — every node gets fully resolved, concrete values. Relative units are converted to pixels, inherited values are filled in, shorthand properties are expanded:

/* What you write */
body { font-size: 16px; color: #333; }
h1   { font-size: 2em; }    /* relative unit */
p    { }                     /* no color set */

/* What the CSSOM stores (computed values) */
/* h1: font-size = 32px   (2 × 16px parent)  */
/* p:  color = rgb(51,51,51)  (inherited from body) */
⚠️
CSS is render-blocking by design The browser will not render a single pixel until all CSS is downloaded and the CSSOM is complete. If a CSS file is slow, the user sees a blank screen. This is why CSS performance matters: keep stylesheets small, eliminate unused CSS, and use media attributes to mark print/non-screen stylesheets as non-blocking.

The Event Loop

The event loop is the mechanism that makes async JavaScript possible without threads. Understanding it means you can predict exactly when any piece of code will run.

┌──────────────────┐    ┌────────────────────────────────────────┐
│   Call Stack     │    │          Heap (Memory)                  │
│                  │    │  Objects, closures, allocated data       │
│  main()          │    └────────────────────────────────────────┘
│  loadData()      │
│  await (paused)  │    ┌────────────────────────────────────────┐
└────────┬─────────┘    │        Web APIs (Browser)               │
         │              │  setTimeout · fetch · DOM events         │
         │              │  Run OUTSIDE the JS call stack           │
         │              └──────────────┬─────────────────────────┘
         │                             │  when done →
         │              ┌──────────────▼─────────────────────────┐
         │              │       Microtask Queue                   │
         │              │  Promise .then · async/await resume     │
         │              │  DRAINS COMPLETELY before next render   │
         │              └────────────────────────────────────────┘
         │
         │              ┌────────────────────────────────────────┐
         └──────────────│       Macrotask Queue                  │
                        │  setTimeout · setInterval · I/O events  │
                        │  ONE task picked per event loop cycle   │
                        └────────────────────────────────────────┘

Event Loop algorithm (repeats forever):
  1. Run call stack to completion (current synchronous code)
  2. Drain ALL microtasks (every Promise .then queued so far)
  3. Render frame (if browser decides a repaint is due)
  4. Pick ONE macrotask from the queue
  5. Go to step 1

Proof with code — predict the output before running it:

console.log('A');                           // synchronous → Call Stack

setTimeout(() => console.log('B'), 0);     // macrotask queue (even at 0ms)

Promise.resolve()
  .then(() => console.log('C'));            // microtask queue

console.log('D');                           // synchronous → Call Stack

// Output order:  A  D  C  B
//
// Step 1: A and D run — they are synchronous, on the call stack
// Step 2: microtasks drain — C runs (Promise .then)
// Step 3: no render needed
// Step 4: one macrotask — B runs (setTimeout)

Why async/await doesn't block:

async function load() {
  console.log('start');                   // synchronous — runs now
  const data = await fetch('/api');       // suspends load(), returns to caller
  console.log('done');                    // resumes here via microtask queue
}

load();
console.log('after call');               // runs while load() is suspended

// Output:  start → after call → done
//
// await = "push the rest of this function into the microtask queue,
//          then give control back to whoever called me"
🔑
Microtasks vs Macrotasks Promises use the microtask queue, which drains completely before the next render or the next setTimeout. This means: chaining ten .then() calls will run all ten before any setTimeout fires — and before the browser gets a chance to repaint. Don't do expensive work in Promise chains without yielding to the event loop occasionally.

Memory, Leaks & Performance

V8 uses mark-and-sweep garbage collection. Starting from "roots" (global scope, call stack, active closures), it follows every reference and marks reachable objects. Everything unreached is swept — freed. Objects are only collected when nothing references them.

Common memory leaks — unintentional references that keep objects alive:

// Leak 1: accidental global variable
function process() {
  result = "oops";   // missing const/let → becomes window.result → never collected
}

// Leak 2: event listener on a removed element
const el = document.querySelector('.modal');
el.addEventListener('click', heavyHandler);
el.remove();
// el is gone from the DOM but heavyHandler holds a reference → not collected

// Fix: use AbortController to remove all listeners at once
const controller = new AbortController();
el.addEventListener('click', handler, { signal: controller.signal });
controller.abort();   // removes all listeners attached with this signal

// Leak 3: closure holding large data unnecessarily
function setup() {
  const bigArray = new Array(100000).fill('data');
  return () => bigArray.length;   // bigArray can never be freed
}

Layout thrashing — the most common performance mistake in DOM manipulation:

// BAD: interleaves reads and writes → forces full reflow each iteration
elements.forEach(el => {
  const h = el.offsetHeight;          // READ → forces layout to finish
  el.style.height = h + 10 + 'px';   // WRITE → invalidates layout
});
// 100 elements = 100 forced reflows

// GOOD: batch all reads first, then all writes
const heights = elements.map(el => el.offsetHeight);   // all reads
elements.forEach((el, i) => {
  el.style.height = heights[i] + 10 + 'px';             // all writes
});
// 1 reflow total

requestAnimationFrame — the right tool for smooth animations:

// BAD: setInterval is not synced to screen refresh
setInterval(() => moveBox(), 16);

// GOOD: rAF is called exactly once per frame, synced to display refresh rate
const startTime = performance.now();

function animate(timestamp) {
  const elapsed = timestamp - startTime;
  box.style.transform = `translateX(${elapsed * 0.2}px)`;

  if (elapsed < 2000) {
    requestAnimationFrame(animate);   // schedule next frame
  }
}

HTTP Headers: The Full Picture

Headers are key-value pairs that carry metadata about a request or response. They control caching, authentication, content negotiation, security, and more — independently of the body.

Request HeaderExample valuePurpose
Hostgithub.comWhich domain to serve — required in HTTP/1.1
Content-Typeapplication/jsonFormat of the request body
Acceptapplication/jsonFormats the client can handle in the response
AuthorizationBearer eyJhbGc...Credentials — API token or JWT
Cookiesession=abc123Cookies the browser automatically sends
Originhttps://myapp.comWhere the request originated from — used by CORS
Response HeaderExample valuePurpose
Content-Typeapplication/json; charset=utf-8Format of the response body
Set-Cookiesession=abc; HttpOnly; SecureTell browser to store a cookie

Caching — how browsers avoid hitting the server again:

First request:
  Browser → Server:  GET /logo.png
  Server → Browser:  200 OK
                     Cache-Control: max-age=86400   ← cache for 24 hours
                     ETag: "v3"

Second request (within 24 hours):
  Browser: serves from local cache — zero network request ← fastest possible

Third request (after 24 hours, ETag revalidation):
  Browser → Server:  GET /logo.png
                     If-None-Match: "v3"      ← "I have version v3, still current?"

  Server → Browser:  304 Not Modified          ← "yes, use your cache"
                     (no body — saves all bandwidth)

Cookie security flags — always set all three:

Set-Cookie: session=abc123; HttpOnly; Secure; SameSite=Strict

HttpOnly      → JavaScript cannot read this cookie (blocks XSS theft)
Secure        → Only sent over HTTPS, never plain HTTP
SameSite=Strict → Not sent on cross-site requests (blocks CSRF attacks)

REST API Design & Conventions

REST is not a protocol — it's an architectural style. Follow its conventions and any developer can use your API without reading documentation. The core rule: URLs name resources (things), HTTP methods name actions (verbs).

GET    /api/users          → list all users
GET    /api/users/42       → get user 42
POST   /api/users          → create a new user
PUT    /api/users/42       → replace user 42 entirely
PATCH  /api/users/42       → update specific fields of user 42
DELETE /api/users/42       → delete user 42

Nested resources:
GET    /api/users/42/posts → all posts by user 42
POST   /api/users/42/posts → create a post for user 42

What breaks the convention:
POST   /api/getUser        ← verb in URL (wrong — use GET /api/users/:id)
GET    /api/deleteUser/42  ← GET must never modify data
POST   /api/user/update    ← use PATCH instead
Status CodeWhen to use it
200 OKSuccessful GET, PATCH, DELETE
201 CreatedSuccessful POST — resource created
204 No ContentSuccessful DELETE — nothing to return
400 Bad RequestMalformed request body or query
401 UnauthorizedNo valid credentials provided
403 ForbiddenCredentials valid but action not allowed
404 Not FoundResource doesn't exist at this URL
409 ConflictDuplicate entry or version mismatch
422 UnprocessableValid JSON but fails validation rules
500 Internal Server ErrorBug on the server — shouldn't happen in production

Consistent request/response shapes:

// POST /api/users — request body
{
  "name": "Alice",
  "email": "alice@example.com",
  "password": "secret123"
}

// 201 Created — response (password never returned)
{
  "id": 42,
  "name": "Alice",
  "email": "alice@example.com",
  "createdAt": "2026-06-23T10:30:00Z"
}

// 422 error — consistent error shape
{
  "error": "Validation failed",
  "details": [
    { "field": "email",    "message": "Must be a valid email address" },
    { "field": "password", "message": "Must be at least 8 characters" }
  ]
}

Pagination — never return unbounded lists:

// GET /api/posts?page=2&limit=20
{
  "data": [ "...20 post objects..." ],
  "pagination": {
    "page": 2,
    "limit": 20,
    "total": 847,
    "totalPages": 43
  }
}

JSON: The Web's Data Format

JSON (JavaScript Object Notation) is the universal language of web APIs. Its syntax looks like JavaScript objects but has stricter rules — and those differences catch developers by surprise.

// JavaScript object — flexible
const user = {
  name: 'Alice',           // single quotes OK
  greet() { },             // functions allowed
  born: new Date(),        // Date objects allowed
  id: undefined,           // undefined allowed
};

// JSON — strict
{
  "name": "Alice",         // double quotes REQUIRED on keys AND string values
                           // no functions — silently omitted
                           // Date becomes string: "2026-06-23T10:30:00.000Z"
                           // undefined is omitted entirely
  "score": 98.6,           // numbers as-is
  "active": true,          // lowercase true/false/null only
  "tags": ["admin"],
  "address": { "city": "Berlin" },
  "deletedAt": null
}
// JS object → JSON string (for sending in request body)
const json = JSON.stringify({ name: "Alice", age: 30 });
// '{"name":"Alice","age":30}'

// Pretty-printed (for logs)
JSON.stringify({ name: "Alice", age: 30 }, null, 2);

// JSON string → JS object (after receiving response)
const user = JSON.parse('{"name":"Alice","age":30}');
user.name;   // "Alice"

// Silent surprises:
JSON.stringify({ fn: () => "hi" });    // '{}'      — function dropped
JSON.stringify({ x: undefined });      // '{}'      — undefined dropped
JSON.stringify({ d: new Date() });     // '{"d":"2026-06-23T...Z"}' — Date → string

Navigating nested API responses:

const data = await fetch('/api/users/42/posts').then(r => r.json());

// data shape:
// {
//   "user": { "id": 42, "name": "Alice" },
//   "posts": [
//     { "id": 1, "title": "First post", "tags": ["js", "web"] },
//     { "id": 2, "title": "Second post", "tags": ["css"] }
//   ],
//   "pagination": { "total": 2 }
// }

data.user.name;                        // "Alice"
data.posts[0].title;                   // "First post"
data.posts[0].tags[1];                 // "web"
data.posts.map(p => p.title);         // ["First post", "Second post"]
data.pagination.total;                 // 2

fetch() Mastery

You know the basics from Phase 1. Now the complete picture: every option, proper error handling, cancellation, and parallel requests.

const response = await fetch('/api/users', {
  method:      'POST',
  headers: {
    'Content-Type':  'application/json',
    'Authorization': 'Bearer eyJhbGc...',
    'Accept':        'application/json'
  },
  body:        JSON.stringify(data),  // only for POST / PUT / PATCH
  credentials: 'include',            // send cookies on cross-origin requests
  signal:      controller.signal     // for cancellation
});
⚠️
The most common fetch() mistake fetch() only throws on network failure (no internet, DNS error). A 404 or 500 response does NOT throw — it resolves successfully with response.ok === false. Always check response.ok. 
// WRONG — silently swallows 404 and 500 errors
const data = await fetch('/api/users').then(r => r.json());

// CORRECT — check response.ok first
const response = await fetch('/api/users');
if (!response.ok) throw new Error(`HTTP ${response.status}`);
const data = await response.json();

A reusable wrapper — write once, use everywhere:

async function apiFetch(url, options = {}) {
  const response = await fetch(url, {
    headers: {
      'Content-Type':  'application/json',
      'Authorization': `Bearer ${getToken()}`,
      ...options.headers                      // allow caller to override
    },
    ...options,
    body: options.body ? JSON.stringify(options.body) : undefined
  });

  if (!response.ok) {
    const err = await response.json().catch(() => ({}));
    throw new Error(err.message || `HTTP ${response.status}`);
  }

  return response.status === 204 ? null : response.json();
}

// Clean call sites
const user    = await apiFetch('/api/users/42');
const newUser = await apiFetch('/api/users', { method: 'POST', body: { name: 'Alice' } });
await apiFetch(`/api/users/${id}`,  { method: 'DELETE' });   // returns null (204)

Cancellation with AbortController:

// Timeout pattern
async function fetchWithTimeout(url, ms = 5000) {
  const controller = new AbortController();
  const timer = setTimeout(() => controller.abort(), ms);
  try {
    const res = await fetch(url, { signal: controller.signal });
    clearTimeout(timer);
    return res.json();
  } catch (e) {
    if (e.name === 'AbortError') throw new Error('Request timed out');
    throw e;
  }
}

// Cancel previous search on new keystroke
let ctrl;
async function search(query) {
  ctrl?.abort();
  ctrl = new AbortController();
  return fetch(`/api/search?q=${query}`, { signal: ctrl.signal }).then(r => r.json());
}

Parallel requests — don't await sequentially when you don't have to:

// Sequential — slow (A finishes, then B starts)
const user  = await fetch('/api/user/1').then(r => r.json());
const posts = await fetch('/api/posts').then(r => r.json());

// Parallel — fast (A and B fire simultaneously)
const [user, posts] = await Promise.all([
  fetch('/api/user/1').then(r => r.json()),
  fetch('/api/posts').then(r  => r.json())
]);

// Promise.allSettled — continues even if one request fails
const results = await Promise.allSettled([
  fetch('/api/critical'),
  fetch('/api/optional')
]);
results[0].status;   // "fulfilled" or "rejected"
results[0].value;    // the resolved value (if fulfilled)

CORS & The Same-Origin Policy

CORS errors feel like server errors — but they are not. They are browser errors. The browser is the enforcer of a security rule called the Same-Origin Policy, and CORS is the mechanism that lets servers grant exceptions to that rule. Once you know who is in charge (the browser), everything else clicks into place.

Why the Same-Origin Policy exists

The Same-Origin Policy is a rule built into every web browser to protect users from a specific class of attack. Here is the concrete threat it blocks:

The attack without the Same-Origin Policy:

  1. You log into bank.com in one tab.
     Your browser now holds a session cookie for bank.com.

  2. In another tab you visit evil.com.
     evil.com's JavaScript starts running in YOUR browser.

  3. evil.com's JS calls:
       fetch('https://bank.com/api/transfer?to=attacker&amount=5000')
     Your browser automatically attaches your bank.com session cookie
     to this request (cookies are sent by origin, not by page).

  4. bank.com's server sees a perfectly valid, authenticated request
     and executes the transfer.

  5. evil.com's JS reads the response — now it knows your balance,
     your account number, or whatever the API returned.

The Same-Origin Policy blocks step 5:
  The browser lets the request go out, receives the response from
  bank.com, then REFUSES to hand that response to evil.com's JavaScript.
  The attacker's script gets a CORS error instead of your data.

This is why the rule exists: without it, any website you visit could silently weaponize your browser against every other site you are logged into.

Origin = protocol + hostname + port. All three must match exactly for two URLs to be considered the same origin:

Your page is at:  https://myapp.com/dashboard

Request target                     Same origin?
https://myapp.com/api/users        ✓  (path never matters)
http://myapp.com/api/users         ✗  different protocol (http vs https)
https://api.myapp.com/users        ✗  different subdomain counts as different host
https://myapp.com:3001/users       ✗  different port
https://othersite.com/api/users    ✗  different domain entirely

Who checks for CORS — and when

The browser is the only entity that enforces the Same-Origin Policy. The server does not know or care about it — it receives a request and responds normally. After the response arrives, the browser inspects the response headers. If the response does not include an Access-Control-Allow-Origin header that matches the page's origin, the browser throws a CORS error and hides the response from your JavaScript.

This is why:

  • CORS errors only happen in browsers — not in curl, Postman, or server-to-server calls (those have no Same-Origin Policy to enforce).
  • The fix is always on the server — you need to add the correct response headers so the browser agrees to deliver the response.
  • The server DID respond successfully. The browser is not broken. It is doing its job.

CORS — how servers explicitly grant cross-origin access:

Simple request (GET, or POST with a plain-text body):

  Browser → Server:  GET /api/data
                     Origin: https://myapp.com        ← browser adds this automatically

  Server → Browser:  200 OK
                     Access-Control-Allow-Origin: https://myapp.com  ← server grants permission

  Browser checks:    header present and matches origin?
                     ✓ yes → deliver response to JS
                     ✗ no  → block response, throw CORS error

─────────────────────────────────────────────────────────────────────

Preflight (POST/PUT/DELETE with JSON body, or custom headers like Authorization):

  The browser first sends a "may I?" request before the real one.
  This lets the server reject dangerous requests before they run.

  Browser → Server:  OPTIONS /api/users
                     Origin: https://myapp.com
                     Access-Control-Request-Method: POST
                     Access-Control-Request-Headers: Authorization

  Server → Browser:  204 No Content
                     Access-Control-Allow-Origin: https://myapp.com
                     Access-Control-Allow-Methods: GET, POST, PATCH
                     Access-Control-Allow-Headers: Authorization, Content-Type
                     Access-Control-Max-Age: 86400    ← browser may cache this for 24 h

  Browser: preflight approved → now sends the actual POST request
🔑
The CORS error is thrown by the browser, not the server The server responded with 200. The browser received the response, looked for Access-Control-Allow-Origin in the headers, did not find one (or found the wrong value), and refused to hand the response to your JavaScript. The network tab shows a 200 — yet your JS gets an error. That is the browser enforcing the rule. The fix is always to add the header on the server. 
// Express server — configuring CORS with the cors package

const cors = require('cors');

// Development: allow all origins (never use in production)
app.use(cors());

// Production: restrict to your frontend's exact origin
app.use(cors({
  origin:         'https://myapp.com',
  credentials:    true,          // required if frontend sends cookies
  methods:        ['GET', 'POST', 'PUT', 'PATCH', 'DELETE'],
  allowedHeaders: ['Content-Type', 'Authorization']
}));

Credentials + CORS — the extra step most tutorials miss:

By default, even after you enable CORS, cookies are not sent cross-origin. You must opt in on both sides — and when you do, Access-Control-Allow-Origin: * is no longer allowed; the browser requires an exact origin match.

// Frontend: tell fetch to include cookies on cross-origin requests
fetch('https://api.myapp.com/user', {
  credentials: 'include'   // without this line, cookies are silently omitted
});

// Backend: two things are now required:
// 1. Access-Control-Allow-Origin must be the exact origin, not '*'
// 2. You must also send Access-Control-Allow-Credentials: true
//
// The cors package handles both when you set credentials: true
// and provide a specific origin (not a wildcard).

Node.js: JavaScript on the Server

Node.js is not a language — it's a runtime. It takes V8 (the same engine Chrome uses) and adds APIs that only make sense outside the browser: file system access, network sockets, the ability to listen for TCP connections.

Browser JS:  V8 engine + Web APIs (DOM, fetch, localStorage, setTimeout...)
Node.js:     V8 engine + Node APIs (fs, http, net, process, Buffer...)

They share the same language. They have completely different standard libraries.

When you run node server.js, Node starts its event loop and your file runs top-to-bottom, registering callbacks. The event loop then waits for I/O — incoming HTTP connections, database responses, file reads — and fires your callbacks when they're ready. One thread, never blocking.

The module system — two flavors:

// ES Modules (add "type": "module" to package.json)

import express from 'express';

Express.js: HTTP Routing & Middleware

Express is a minimal routing and middleware framework. At its core: match an incoming URL + method → run a handler → send a response. Every incoming request flows through a pipeline:

Incoming request
      ↓
  Middleware 1 (CORS headers)
      ↓
  Middleware 2 (parse JSON body)
      ↓
  Middleware 3 (logging)
      ↓
  Route handler  ←  GET /api/users/:id
      ↓
  Send response

Bare minimum server:

const express = require('express');
const app = express();

app.use(express.json());   // parse JSON request bodies — must come first

app.get('/', (req, res) => {
  res.json({ message: 'Hello world' });
});

app.listen(3000, () => console.log('Server on :3000'));

req and res — what's inside:

app.post('/api/users/:id', (req, res) => {
  // req — the incoming request
  req.method           // 'POST'
  req.params.id        // route param: /api/users/42 → '42'
  req.query.page       // query string: ?page=2 → '2'
  req.body             // parsed JSON body (requires express.json())
  req.headers['authorization']

  // res — build the response
  res.status(201).json({ id: 1, name: 'Alice' });  // status + JSON
  res.status(404).json({ error: 'Not found' });
  res.sendStatus(204);                              // body-less (DELETE)
});

Middleware — the power pattern:

// Custom logging middleware
app.use((req, res, next) => {
  console.log(`${req.method} ${req.url}`);
  next();   // ← MUST call next() or requests hang forever
});

// Error-handling middleware — 4 params, always LAST
app.use((err, req, res, next) => {
  console.error(err.stack);
  res.status(500).json({ error: 'Something went wrong' });
});

Router — split routes into separate files:

// routes/users.js
const router = require('express').Router();

router.get('/',       listUsers);
router.get('/:id',    getUser);
router.post('/',      createUser);
router.patch('/:id',  updateUser);
router.delete('/:id', deleteUser);

module.exports = router;

// src/index.js — mount routers at a base path
app.use('/api/users', require('./routes/users'));
app.use('/api/posts', require('./routes/posts'));
ℹ️
When a route is mounted at /api/users, the router sees paths relative to that prefix. So router.get('/:id', ...) matches /api/users/42, not just /42.

PostgreSQL & SQL

A relational database stores data in tables (rows + columns) and enforces relationships via foreign keys. It guarantees ACID: data is never half-written, never corrupt — even if the server crashes mid-write.

Table: users
┌────┬─────────┬───────────────────────┬─────────────────────┐
│ id │ name    │ email                 │ created_at          │
├────┼─────────┼───────────────────────┼─────────────────────┤
│  1 │ Alice   │ alice@example.com     │ 2026-01-15 10:00:00 │
│  2 │ Bob     │ bob@example.com       │ 2026-02-20 14:30:00 │
└────┴─────────┴───────────────────────┴─────────────────────┘

Table: posts
┌────┬─────────┬─────────────────────────┐
│ id │ user_id │ title                   │
├────┼─────────┼─────────────────────────┤
│  1 │       1 │ My first post           │  ← Alice's post
│  2 │       2 │ Hello from Bob          │  ← Bob's post
└────┴─────────┴─────────────────────────┘
         ↑ FOREIGN KEY — links to users.id
-- Create tables
CREATE TABLE users (
  id         SERIAL PRIMARY KEY,       -- auto-incrementing integer
  name       VARCHAR(100) NOT NULL,
  email      VARCHAR(255) UNIQUE NOT NULL,
  password   TEXT NOT NULL,
  created_at TIMESTAMPTZ DEFAULT NOW()
);

CREATE TABLE posts (
  id      SERIAL PRIMARY KEY,
  user_id INTEGER REFERENCES users(id) ON DELETE CASCADE,
  title   VARCHAR(255) NOT NULL,
  body    TEXT
);

-- CRUD
SELECT id, name FROM users WHERE id = 1;
INSERT INTO users (name, email, password) VALUES ('Alice', 'a@b.com', 'hash') RETURNING id, name;
UPDATE users SET name = 'Alice Smith' WHERE id = 1;
DELETE FROM posts WHERE id = 3;

-- JOIN — combine two tables in one query
SELECT posts.title, users.name AS author
FROM posts
JOIN users ON posts.user_id = users.id
ORDER BY posts.id DESC
LIMIT 10;

Using PostgreSQL from Node (the pg package):

// db.js — connection pool (reuse connections, never open one per query)
const { Pool } = require('pg');
const pool = new Pool({ connectionString: process.env.DATABASE_URL });
module.exports = pool;

// In a route handler
const pool = require('./db');

app.get('/api/users/:id', async (req, res, next) => {
  try {
    const { rows } = await pool.query(
      'SELECT id, name, email FROM users WHERE id = $1',
      [req.params.id]    // ← parameterized — NEVER string-concatenate
    );
    if (!rows.length) return res.status(404).json({ error: 'Not found' });
    res.json(rows[0]);
  } catch (err) {
    next(err);           // pass to error-handling middleware
  }
});
⚠️
SQL injection — never concatenate user input into queries 
// VULNERABLE — attacker can destroy your entire database
pool.query(`SELECT * FROM users WHERE email = '${req.body.email}'`);

// SAFE — $1 is always treated as data, never as SQL syntax
pool.query('SELECT * FROM users WHERE email = $1', [req.body.email]);

Authentication & JWT

Authentication answers "who is this user?" Authorization answers "what are they allowed to do?" — always in that order.

You never store passwords. You store their hash. A hash is a one-way function. bcrypt adds a random salt per password and is deliberately slow to defeat brute-force attacks:

const bcrypt = require('bcryptjs');

// Registration — hash before storing
const hash = await bcrypt.hash(req.body.password, 12);  // 2^12 iterations ≈ 100ms
await pool.query(
  'INSERT INTO users (email, password) VALUES ($1, $2)',
  [req.body.email, hash]
);

// Login — compare submitted password against stored hash
const { rows } = await pool.query(
  'SELECT * FROM users WHERE email = $1', [req.body.email]
);
if (!rows.length) return res.status(401).json({ error: 'Invalid credentials' });

const match = await bcrypt.compare(req.body.password, rows[0].password);
if (!match)   return res.status(401).json({ error: 'Invalid credentials' });
// ↑ Same error message both cases — don't tell an attacker which field was wrong

JWT — stateless authentication. No database lookup needed to verify.

A JWT has three base64url-encoded parts, joined by dots:

eyJhbGciOiJIUzI1NiJ9 . eyJpZCI6NDIsImVtYWlsIjoiYUBiLmNvbSIsImlhdCI6MTcxOX0 . 8xQ4kA...
         ↑                                    ↑                                       ↑
      Header                              Payload                              Signature
    {alg:"HS256"}              {id:42, email:"a@b.com", iat:1719...}       HMAC-SHA256
                               (base64url — NOT encrypted, readable)     (only server can forge)
const jwt = require('jsonwebtoken');

// After login — sign and issue a token
const token = jwt.sign(
  { id: user.id, email: user.email },   // payload — keep small, no passwords
  process.env.JWT_SECRET,               // signing secret — 32+ random chars
  { expiresIn: '24h' }
);
res.json({ token });

// Middleware — protect routes that require login
function requireAuth(req, res, next) {
  const header = req.headers.authorization;
  if (!header?.startsWith('Bearer ')) {
    return res.status(401).json({ error: 'No token provided' });
  }
  try {
    const decoded = jwt.verify(header.slice(7), process.env.JWT_SECRET);
    req.user = decoded;   // attach decoded payload to request
    next();
  } catch {
    res.status(401).json({ error: 'Invalid or expired token' });
  }
}

// Apply to protected routes
app.get('/api/me', requireAuth, async (req, res) => {
  const { rows } = await pool.query(
    'SELECT id, name, email FROM users WHERE id = $1', [req.user.id]
  );
  res.json(rows[0]);
});

The complete auth flow end-to-end:

1
Register — POST /api/auth/register { email, password }
Server validates → bcrypt.hash → INSERT INTO users → 201 { id, email }
2
Login — POST /api/auth/login { email, password }
SELECT user → bcrypt.compare → jwt.sign → 200 { token: "eyJ..." }
3
Client stores token — typically in memory or localStorage
Every subsequent request adds: Authorization: Bearer eyJ...
4
Protected route — GET /api/me
requireAuth middleware: jwt.verify(token, secret) → attach req.user → handler runs → 200 { name, email }

5.1

How Frontend and Backend Talk

You've built frontends and backends separately. The integration challenge is that they must agree on a contract — the API shape — and nothing enforces that contract except discipline.

Two deployment models:

Model A — Separate origins (most common in production)
  Frontend:  https://myapp.vercel.app    (HTML/CSS/JS, hosted on Vercel)
  Backend:   https://myapi.railway.app   (Express, hosted on Railway)
  → Different domains → CORS required
  → Frontend fetches backend over the internet

Model B — Same origin (Express serves everything)
  https://myapp.com/          → Express serves index.html
  https://myapp.com/api/users → Express handles the API
  → No CORS needed. One deployment. Simpler for small projects.

Same-origin setup — Express serving the frontend:

const path = require('path');

// Serve HTML/CSS/JS from a /public folder
app.use(express.static(path.join(__dirname, '../public')));

// API routes still work normally
app.use('/api/auth',  require('./routes/auth'));
app.use('/api/posts', require('./routes/posts'));

// Fallback: unknown routes → serve index.html (for client-side routing)
app.get('*', (req, res) => {
  res.sendFile(path.join(__dirname, '../public', 'index.html'));
});

The API contract — write this before building either side:

POST /api/auth/login
  Request:  { "email": string, "password": string }
  Success:  200 { "token": string, "user": { "id", "name", "email" } }
  Error:    401 { "error": "Invalid credentials" }

GET /api/posts   (Authorization: Bearer <token>)
  Success:  200 { "data": Post[], "pagination": { "total": number } }
  Error:    401 { "error": "No token provided" }

POST /api/posts  (Authorization: Bearer <token>)
  Request:  { "title": string, "body": string }
  Success:  201 Post
  Error:    422 { "error": "Title and body are required" }
🔑
Changing the API contract mid-build means updating two codebases simultaneously. Write it down first. Treat it as a commitment.
5.2

State Management in the Frontend

When your app has a session and server data, you need a clear answer to: where does this piece of information live?

Where to keep the JWT:

// Option A: localStorage — simple, survives refresh, readable by JS
localStorage.setItem('token', token);
const token = localStorage.getItem('token');
localStorage.removeItem('token');

// Option B: memory variable — gone on refresh, immune to XSS
let _token = null;
function setToken(t) { _token = t; }
function getToken()  { return _token; }

// Option C: HttpOnly cookie (set by server) — best security
// Server: res.cookie('token', jwt, { httpOnly: true, secure: true, sameSite: 'strict' })
// Browser: sends it automatically; JS cannot access it at all
ℹ️
For a beginner project, localStorage is fine. For production with sensitive data, HttpOnly cookies are the correct choice — they're invisible to XSS attacks entirely.

A minimal state pattern — the core idea behind every framework:

// Single source of truth
const state = {
  user:    null,   // { id, name, email } or null
  posts:   [],
  loading: false,
  error:   null,
};

// One function to update state + trigger re-render
function setState(patch) {
  Object.assign(state, patch);
  render();
}

// One function to update the DOM from state
function render() {
  // Auth state
  document.getElementById('auth-section').hidden = !!state.user;
  document.getElementById('post-form').hidden    = !state.user;
  document.getElementById('user-info').textContent =
    state.user ? `Logged in as ${state.user.name}` : '';

  // Posts list
  const list = document.getElementById('posts-list');
  if (state.loading) {
    list.innerHTML = '<p class="loading">Loading...</p>';
  } else if (state.error) {
    list.innerHTML = `<p class="error">${state.error}</p>`;
  } else {
    list.innerHTML = state.posts
      .map(p => `<article><h3>${p.title}</h3><p>${p.body}</p></article>`)
      .join('');
  }
}

This pattern — one state object, one render function, explicit updates — is the core concept behind React, Vue, and Svelte. They make it faster and handle edge cases, but the mental model is identical.

// src/routes/posts.js
const router      = require('express').Router();
const pool        = require('../db');
const requireAuth = require('../middleware/auth');

// Public — anyone can read posts
router.get('/', async (req, res, next) => {
  try {
    const { rows } = await pool.query(`
      SELECT posts.id, posts.title, posts.body, posts.created_at,
             users.name AS author
      FROM posts
      JOIN users ON posts.user_id = users.id
      ORDER BY posts.created_at DESC LIMIT 20
    `);
    res.json({ data: rows });
  } catch (err) { next(err); }
});

// Protected — only logged-in users can create posts
router.post('/', requireAuth, async (req, res, next) => {
  const { title, body } = req.body;
  if (!title?.trim() || !body?.trim())
    return res.status(422).json({ error: 'Title and body are required' });
  try {
    const { rows } = await pool.query(
      'INSERT INTO posts (user_id, title, body) VALUES ($1, $2, $3) RETURNING *',
      [req.user.id, title.trim(), body.trim()]
    );
    res.status(201).json(rows[0]);
  } catch (err) { next(err); }
});

The frontend — apiFetch wrapper + actions + event listeners:

// public/app.js

const API = '';   // '' = same origin; or 'https://myapi.railway.app'

// Auth token helpers
function getToken()  { return localStorage.getItem('token'); }
function setToken(t) { localStorage.setItem('token', t); }
function clearToken(){ localStorage.removeItem('token'); }

// Reusable fetch wrapper
async function apiFetch(path, options = {}) {
  const token = getToken();
  const res = await fetch(API + path, {
    headers: {
      'Content-Type': 'application/json',
      ...(token ? { Authorization: `Bearer ${token}` } : {})
    },
    ...options,
    body: options.body ? JSON.stringify(options.body) : undefined
  });
  if (!res.ok) {
    const err = await res.json().catch(() => ({}));
    throw new Error(err.error || `HTTP ${res.status}`);
  }
  return res.status === 204 ? null : res.json();
}

// Actions
async function login(email, password) {
  const { token, user } = await apiFetch('/api/auth/login', {
    method: 'POST', body: { email, password }
  });
  setToken(token);
  setState({ user });
}

async function logout() {
  clearToken();
  setState({ user: null });
}

async function loadPosts() {
  setState({ loading: true, error: null });
  try {
    const { data } = await apiFetch('/api/posts');
    setState({ posts: data, loading: false });
  } catch (err) {
    setState({ error: err.message, loading: false });
  }
}

async function createPost(title, body) {
  await apiFetch('/api/posts', { method: 'POST', body: { title, body } });
  await loadPosts();
}

// Bootstrap on page load
(async function init() {
  await loadPosts();
  if (getToken()) {
    try {
      const user = await apiFetch('/api/me');
      setState({ user });
    } catch {
      clearToken();   // expired or invalid — start fresh
    }
  }
})();

Error Handling End to End

Most bugs in full-stack apps happen when errors are swallowed silently at one layer and the next layer receives undefined. Errors need a clean path from the database all the way to a user-visible message.

Database error (e.g. unique violation: email already exists)
    ↓  caught in route try/catch
    ↓  code '23505' detected → res.status(409).json({ error: 'Email taken' })
    ↓  fetch() gets 409 response
    ↓  apiFetch(): response.ok is false → throws Error('Email taken')
    ↓  caller's catch block → setState({ error: 'Email taken' })
    ↓  render() reads state.error → shows message in DOM
    ↓
User sees: "That email is already taken"

Backend validation — catch bad input before it hits the database:

router.post('/register', async (req, res, next) => {
  const { name, email, password } = req.body;

  const errors = [];
  if (!name?.trim())                errors.push('Name is required');
  if (!email?.includes('@'))        errors.push('Valid email required');
  if ((password?.length ?? 0) < 8) errors.push('Password must be 8+ chars');
  if (errors.length)
    return res.status(422).json({ error: errors.join(', ') });

  try {
    const hash = await bcrypt.hash(password, 12);
    const { rows } = await pool.query(
      'INSERT INTO users (name, email, password) VALUES ($1, $2, $3) RETURNING id, name, email',
      [name.trim(), email.toLowerCase(), hash]
    );
    res.status(201).json(rows[0]);
  } catch (err) {
    if (err.code === '23505')   // PostgreSQL unique violation
      return res.status(409).json({ error: 'That email is already taken' });
    next(err);
  }
});

The Mental Model: Request → Response → Render

When something doesn't work in a full-stack app, this flow tells you exactly which layer to investigate:

1
Frontend validation — Did the input pass client-side checks? If not, the request never fires. Check the form logic.
2
Request sent? — Open DevTools → Network tab. Find the request. Did it fire? If not, check the fetch() call and event listener.
3
Express received it? — Check server logs. Did the route match? Did middleware run (CORS, express.json, requireAuth)?
4
Database query? — Did the pg query succeed? Check pg error codes. Is the connection string correct?
5
Response shape? — Back in the Network tab: check the status code and response body. Is it what the frontend expects?
6
State updated? — Did setState() get called with the right data? Did render() run? Check with console.log(state).
🔑
The golden rule of full-stack debugging: open DevTools → Network → find the failing request → read the response body. That one action answers 80% of full-stack bugs.

Before:  http://myapp.com   → plain text, no padlock, HTTP/1.1
